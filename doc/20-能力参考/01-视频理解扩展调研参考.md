# 01-视频理解扩展调研参考

## 1. 总体判断

对 `pi-sense`（由 `pi-dvision` 扩展而来）的视频理解能力，采用**双路线按需求分流**（详见 `doc/决策档案/0001`）：

- **原生视频模型路线**（内容理解 / what）：借鉴 `pi-multimodal-proxy` / `pi-web-access` 的“直接传视频给原生支持视频的模型”思路。当前实现里真实接通的是 MiniMax-M3；Grok / Gemini 仍只保留 provider 分发接缝与参数契约。原生路线快、内容识别准，但模型时间感知不可靠。
- **抽帧 + 转录 + vision model 描述路线**（时间感知 / when）：借鉴 `pi-watch` + `507-breakdown`，插件自己控制帧时间点和 ASR 时间戳，时间-内容对应可靠。

两路线不是主/fallback，而是按用户意图分流：原生路线负责“what”，抽帧路线负责“when”。

## 2. 核心机制速览

### 2.1 `pi-multimodal-proxy` — 原生 video model + wire-format hack

- **检测**：在 `before_agent_start` 里扫描 prompt 文本，正则匹配 `.mp4`、`.mov` 等路径，读取为 base64。
- **载体 hack**：把视频包装成 `PiAiImage` 形状 `{ type: "image", data, mimeType: "video/mp4" }`。
- **格式改写**：通过 `complete()` 的 `onPayload` hook 调用 `fixVideoAudioPayload()`，把 pi-ai 序列化后的 `image_url` 改回 `video_url`。
- **模型调用**：默认用 `xai/grok-4.3` 或 Gemini 等原生支持视频的模型；对 xAI 还专门拆分了 STT 与 Responses File 两条路径。
- **结果注入**：把模型返回的描述包成 `<vision_proxy_video_description>` fence 注入 system prompt。

### 2.2 `pi-web-access` — Gemini 原生视频理解

- **本地视频**：`isVideoFile()` 检测路径 → 上传 Gemini Files API → 轮询 ACTIVE → 调用 Gemini 分析 → 删除云端文件。
- **YouTube 视频**：先尝试 Gemini Web（浏览器 cookie），失败 fallback 到 Gemini API，再失败 fallback 到 Perplexity 总结。
- **输出**：直接返回 markdown 格式 `ExtractedContent`（title + content + thumbnail）。

### 2.3 `@artale/pi-watch` — 抽帧 + 转录 + Claude

- **下载**：`yt-dlp` 下载视频/字幕。
- **抽帧**：`ffmpeg` 按比例抽帧（默认上限 100 帧，按时长动态预算）。
- **转录**：优先用 YouTube 免费字幕，否则 Groq Whisper 转录音频。
- **交给 Claude**：把帧图片 + 转录文本一起给 Claude 分析。

### 2.4 `507-breakdown` — 视频拉片取证工作流

- **文字证据链四级降级**：平台原生字幕 → 平台自动字幕 → 本地 ASR → 帧内 OCR。
- **视觉双采样**：`1 fps` 均匀抽帧 + scene-cut 场景切换抽帧 + 联系图。
- **两层分段**：分析层按 6-8 秒小窗处理证据，输出层按语义合并。
- **产物**：结构化 `manifest.json` + `transcript.md` + `breakdown.md` + `breakdown.json`。

## 3. 与 `pi-sense` 的关系

### 3.1 设计冲突（明确不借）

> 注：本节最初写于 grill 阶段，后被 507 的 MiniMax 实测推翻。当前正式决策以 ADR `0001-视频理解双路线-原生内容理解与抽帧时间感知分流` 为准：原生视频路线已纳入 `pi-sense`，但只负责 `what`，不负责可信 `when`。

1. **`pi-multimodal-proxy` 的高级功能**
   - 理由：`analyze_image` 工具、crop、session recall、compaction digest、grounding 坐标、folders allowlist、pre-consented providers 等，与 `pi-dvision` AGENTS.md 中"保持最小实现"、"不为跨 provider 兼容提前做复杂抽象"冲突。
   - 影响范围：第一版不做这些能力，仅保留"描述 + 注入"核心。

### 3.2 同思路（已有，印证方向）

1. **在 `tool_result` / `context` 注入描述文本**
   - `pi-dvision` 已经在做，`pi-multimodal-proxy` 同样走拦截注入路线，印证这个架构可行。
   - 影响范围：视频理解的描述也走同样注入点，无需新增 hook。

2. **按 hash 缓存描述**
   - `pi-dvision` 已有 image hash 缓存，`pi-multimodal-proxy` 同样按 hash 缓存并支持 recall。
   - 影响范围：视频可复用缓存思想，但第一版可先按视频文件 hash 缓存转录+描述结果。

### 3.3 候选（值得吸收）

| # | 候选机制 | 来源 | 改造点 |
|---|---|---|---|
| 1 | 视频文件路径/附件检测 | `pi-multimodal-proxy` | 在 `index.ts` 增加 `extractVideoFromBlock()` / `extractVideoPaths()`，复用现有 block 解析结构 |
| 2 | 抽帧 + 转录流水线 | `pi-watch` + `507-breakdown` | 新增 `video-processor.ts`：用 `ffmpeg` 抽帧、`faster-whisper` / `whisper.cpp` 转录，再把帧交给现有 `describeImage()` |
| 3 | 文字证据链四级降级 | `507-breakdown` | 改音频处理：平台原生字幕 → 自动字幕 → 本地 ASR → 帧内 OCR（必要时） |
| 4 | 视觉双采样 | `507-breakdown` | 均匀抽帧 + scene-cut 场景切换抽帧，而不是固定 fps |
| 5 | 动态帧预算 | `pi-watch` | 按视频时长计算帧数：≤30s 30帧，>10min 100帧 |
| 6 | YouTube URL 支持 | `pi-web-access` + `pi-watch` | 新增 `youtube.ts`：检测 URL，用 `yt-dlp` 下载/取流，再交给 video-processor |
| 7 | 图像/视频模型可分别配置 | `pi-multimodal-proxy` | 扩展 `DvisionConfig` → `SenseConfig`，增加 `videoModel`、`asrProvider` 等字段，但保持最小 |
| 8 | 文件访问白名单/数据外泄提示 | `pi-multimodal-proxy` | 扩展视频路径校验，只允许 cwd/tmp/用户显式目录；在首次发送视频到外部模型时提示数据外泄 |
