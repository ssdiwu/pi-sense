# 视频理解扩展调研参考

本页记录影响 `pi-sense` 视频设计的外部实现。它不是功能承诺；当前产品能力以根目录 `README.md` 和 ADR 0001 为准。

## 结论

外部项目印证了两件事：

1. 原生视频模型适合整段内容理解，但不同 provider 的上传、媒体引用和请求格式差异很大。
2. 抽帧与转录能提供可追溯的时间证据，适合回答时间点与顺序问题。

因此 `pi-sense` 采用 ADR 0001 定义的双路线：原生路线处理 what，抽帧 + ASR 路线处理 when。

## 参考实现

### pi-multimodal-proxy

**可借鉴**：以 image-shaped carrier 通过通用 SDK，并在 `onPayload` 时改写为 provider 所需的视频 payload；按 hash 缓存媒体描述。

**没有采用**：crop、grounding 坐标、复杂 session recall、目录白名单与跨 provider 大型抽象。这些超出 `pi-sense` 的“描述 + 注入”职责。

### pi-web-access

**可借鉴**：Gemini Files API 上传、等待文件可用、调用模型并清理远程文件的完整流程。

**没有采用**：YouTube URL、浏览器 cookie、Perplexity fallback。第一版仅支持本地视频路径。

### @artale/pi-watch

**可借鉴**：按视频时长限制帧预算，并把帧与带时间戳的转录一并交给模型分析。

**差异**：`pi-sense` 不下载远程视频；时间路线使用本机 `ffmpeg` 和本地 ASR，帧上限固定为 120。

### 507-breakdown

**可借鉴**：文字证据优先、均匀抽帧与场景切换采样、用时间小窗组织视觉与音频证据。

**没有采用**：拉片包、联系图、结构化分析产物。这些属于视频分析工作流，不属于实时 Pi handoff。

## 当前采用范围

- MiniMax-M3 的 Files API 上传与 `mm_file://` 媒体引用。
- `complete()` 的 `onPayload` 改写，以适配 provider 特定的 video block。
- 本地视频 hash 与用户问题 hash 缓存。
- ≤ 1 分钟按 0.5 秒间隔抽帧；更长视频在配置的帧预算内均匀采样（默认 120，最大 600）。
- `whisper-cli` 优先、faster-whisper 回退的本地 ASR。

## 后续评估条件

仅当某项能力解决可复现的用户问题时才考虑加入，例如：某 provider 的原生 adapter 完成真实联调，或实际视频任务证明固定采样遗漏关键事件。
