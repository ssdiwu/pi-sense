# 0001-视频理解双路线-原生内容理解与抽帧时间感知分流

## 背景

507 实测 MiniMax-M3 原生视频理解可用（`video_url` + `mm_file://` 上传，fps 0.2-5），且 `~/.pi/agent/pi-sense.json` 配置的 visionModel 正是 `minimax-cn/MiniMax-M3`。但实测暴露一个关键问题：**原生模型的时间感知不可靠**——一段 10.3 秒的视频，模型把动作时间理解成 0.0s-1.1s，时间戳和真实时长系统性偏差。

同时，grill 阶段曾锁定"不走原生 video model，只做抽帧+ASR"的路线。507 的实测推翻了这个判断：原生路线在"内容理解"上又快又准，不该放弃。

## 决定

视频理解走**双路线按需求分流**，不是简单的"主/fallback"：

- **原生视频模型路线**：用于"理解视频内容"（what）——总结、描述、回答关于内容的问题。模型直接看整段视频，快、简单、内容识别准。
- **抽帧 + 本地 ASR 路线**：用于"时间感知"（when）——用户需要精确时间戳（"1:30 发生了什么"、"哪个时间点出现 X"、时间线）时，插件自己用 ffmpeg 抽帧（每帧带已知时间点）+ 本地 ASR（带时间戳的 segment），建立可靠的时间-内容对应。

## 为什么

- 原生模型内容理解准但时间感知不可靠：实测能把拖拽动作识别出来（内容对），但时间戳和真实时长对不上（时间错）。
- 抽帧+ASR 路线我们控制每个帧的采样时间点和 ASR segment 的时间戳，所以时间-内容对应是确定性的、可追溯的。
- 两条路线各有不可替代的优势：原生路线的速度和内容理解力，抽帧路线的时间可靠性。单走任何一条都有明显短板。

## 影响

- `SenseConfig` 需要同时支持两套参数：原生路线的 `videoFps`（0.2-5）、`videoThinking`（思考开关）；抽帧路线的 `maxVideoFrames`、`asrProvider`。
- 需要一个 provider 视频兼容层（MiniMax/Grok/Gemini 各自的上传 + payload 改写），原生路线用它。
- 分流方式：auto（检测用户 prompt 里的时间相关意图）或显式配置，待实现时细化。
