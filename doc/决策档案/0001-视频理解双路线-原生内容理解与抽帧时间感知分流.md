# 0001：视频理解双路线——原生内容理解与抽帧时间感知分流

- **状态**：已采纳
- **日期**：2026-07-09

## 背景

视频问题包含两类不同要求：

- **内容理解（what）**：视频讲什么、用户做了哪些操作、画面中有哪些对象。
- **时间感知（when）**：第几秒发生了什么、事件先后顺序、拖动方向和时间线。

原生视频模型对内容理解的质量和效率较好，但其生成的时间戳与源视频真实时长不稳定。单纯抽帧 + ASR 虽能提供可追溯的时间证据，却会增加处理成本，也不应替代原生模型对整体内容的理解。

## 决定

视频处理按问题意图分流，而不是采用“主路线 + fallback”模型：

| 路线 | 负责的问题 | 证据来源 |
|---|---|---|
| 原生视频模型 | 内容、总结、主题、操作概览（what） | provider 对整段视频的理解 |
| 抽帧 + 本地 ASR | 时间点、顺序、拖动方向、时间线（when） | 已知采样时间点的帧和带时间戳 ASR segment |

`videoRoute=auto` 检测用户问题中的时间意图；用户也可显式选择 `native` 或 `frames`。当前原生路线真实接通 MiniMax-M3，其他 provider 不可用时降级到抽帧路线。

## 后果

### 正面

- 内容问题获得原生视频模型的整体理解能力。
- 时间问题由本地可追溯证据支撑，不依赖模型虚构或漂移的时间戳。
- 两条路线的成本与性能随问题类型匹配。

### 代价

- 配置需要同时维护原生参数（`videoFps`、`videoThinking`）和本地时间路线参数（`maxVideoFrames`、`asrProvider`）。
- 时间路线依赖本机 `ffmpeg` 与 ASR 工具，速度通常低于原生路线。
- provider 原生视频协议不同，adapter 必须保持显式且按 provider 验证。

## 不做

- 不把原生模型结果升级为时间事实。
- 不因当前只有 MiniMax 可用而伪造 Gemini/Grok 的兼容承诺。
- 不在第一版实现远程视频下载、实时捕获或自适应补帧。
